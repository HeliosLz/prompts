# 重构 Pipeline

这个 pipeline 只处理真实重构。目标不是把代码换个写法，而是在不改变用户可见行为的前提下，消除错误的数据边界、特殊情况和旧代码残留。

**上下文链接（至少一条）**：
- 相关：[[knowledge/taste-review/content]]

---

## 硬规则

1. 充分相信上游数据。如果缺失数据，应该在上游提供，而不是在下游打补丁。
2. 特别大型的代码块必须分 2-3 个步骤执行；每个步骤结束都要有可体验内容，而不是只产生孤岛代码。新代码生效时，老代码必须删除。
3. 不写兼容、回退、临时、备用、特定模式生效的代码。消除特殊情况，而不是包住特殊情况。
4. 保持当前用户可见行为不变。

---

## Task Blueprint（按顺序执行）

### Task 1：确认这是真问题

**目标**：拒绝为了“更好看”而重构。

**执行步骤**：
1. 指出当前真实问题：数据所有权错误、重复逻辑、边界泄漏、旧路径残留、还是复杂度已经挡住后续功能。
2. 明确不重构会造成的实际成本。
3. 如果说不出一句话的问题定义，停止，不进入实现。

**完成标准**：一句话说明重构目的，并能对应到真实代码路径。

---

### Task 2：钉住数据结构和上游权威

**目标**：先修数据边界，再动控制流。

**执行步骤**：
1. 找出核心数据是什么、谁创建、谁拥有、谁消费。
2. 如果下游缺字段或靠猜测工作，把字段补到上游权威路径。
3. 删除为缺失数据准备的下游补丁分支。

**完成标准**：数据来源单一，下游只消费明确合同，不猜测、不兜底。

---

### Task 3：把大改切成 2-3 个可体验步骤

**目标**：每一步都能运行、能看见效果、能回归验证。

**执行步骤**：
1. 把重构拆成最多 3 个步骤。
2. 每个步骤必须写清：
   - 用户能体验到什么仍然工作
   - 哪条旧代码会在新代码生效时删除
   - 用什么命令或手动路径验证
3. 不允许先堆一批无人使用的新代码。

**完成标准**：每一步结束都不是孤岛代码，且旧路径同步减少。

---

### Task 4：实现时消除特殊情况

**目标**：让普通路径覆盖所有正常输入。

**执行步骤**：
1. 优先改数据结构或调用边界，而不是增加 `if`。
2. 遇到兼容、回退、临时、备用、特定模式分支时，先问为什么普通路径不能处理。
3. 如果旧路径不再是权威路径，直接删除。

**完成标准**：特殊分支减少，旧代码没有继续挂在运行时路径上。

---

### Task 5：验证用户可见行为不变

**目标**：证明重构没有破坏现有用户工作流。

**执行步骤**：
1. 跑与改动范围匹配的 lint/typecheck/test/e2e。
2. 对用户可见路径做手动或自动验证。
3. 如果行为必须变化，先停下来说明原因；这已经不是纯重构。

**完成标准**：当前用户可见行为保持不变，验证结果可复述。

---

## 失败处理

1. 找不到真实问题：不重构。
2. 缺数据只能靠下游猜测：先补上游，不写补丁。
3. 大改无法拆出可体验步骤：缩小范围，先做能落地的一步。
4. 需要保留新旧两套运行时路径：重新设计边界，直到可以删除旧路径。
5. 用户可见行为变化：停止，把任务改为功能变更或破坏性迁移，不伪装成重构。
